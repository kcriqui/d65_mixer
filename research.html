How show software drives 4+ color projectors — and what it means for a beam rig

A grounding document for the d65_mixer project: the color pipeline behind Pangolin BEYOND and its rivals, the ILDA/DAC facts that constrain multi-primary projectors, and the color science that actually governs aerial beams.

Compiled July 2026·Sources: ILDA standards, Pangolin & LaserAnimation manuals, vendor docs, practitioner forums

The one fact that explains everything

24-bit RGB (file / live) → software color engine (per-projector calibration + mix) → N physical channels (3–6) → DAC → ILDA pins → laser modules

A 4th, 5th, or 6th color is never stored in the show file. The file format is RGB-only; the extra primary is a DAC output channel whose drive level the software computes from RGB using a proprietary, calibration-based mapping. There is no cross-vendor standard for it — that missing mapping is the whole game for multi-color quality.

01

The pipeline, and why the 4th color lives in the DAC

Every package — QuickShow, BEYOND, Lasergraph DSP, Phoenix, LSX — shares the same chain. The artwork (an .ild frame or a live-generated shape) carries only RGB. The software holds that as a 24-bit image, then a color engine decides how much of each physical laser channel to emit, per projector, from a stored calibration. The DAC turns those channel values into analog voltages on the ILDA connector, and the projector's diode drivers follow.

So a projector with a real yellow laser doesn't get "yellow" from the file — the software derives the yellow channel from the RGB image plus a calibration profile, and can choose to fire the dedicated yellow at 100% instead of summing red + green. The extra primary is a dedicated output at the DAC/pin level; its value is computed, not transmitted.

The practical trap. A plain 3-wire RGB ILDA path physically cannot light a 4th laser. To use extra primaries you must wire those diode drivers to the DAC's own extra-color outputs (FB4 Color 4/5/6, or the DSP's) — the software's multi-channel math has nowhere to go otherwise.

Two ILDA standards, doing two different jobs

Color is governed by two independent ILDA specs. Conflating them is the source of most confusion.

The file format (.ild / IDTF) — stores intent, RGB only documented

Every mode expresses color as RGB or an RGB-valued palette — no field for a yellow/cyan/deep-red laser level. Formats 4/5 (true color) were added later; older references list only 0/1/2, which is why "someone else's file loads with wrong colors" (a mismatched default palette) is a classic complaint.

Only R, G, B are mandatory, all analog 0–5 V (not TTL). The three extra color pins are formally "user-defined" but the industry has settled on deep-blue / yellow / cyan. There is no reserved pin for deep red — it has to reuse a user channel. Net ceiling: 6 color wires.

How BEYOND does it — and its one real limit

BEYOND's Advanced Color is a palette-training system: you set per-channel "Color Magnitude" sliders at a handful of trained hues (the six primaries + white, plus custom hues you add like orange), and it interpolates between them onto up to six FB4 channels. Per-channel analog-linear/log modes, multi-point gamma, warm↔ice white balance, and separate beam-vs-graphics palettes are all there.

The limit: it's an RGB→N-channel remap driven by manual training, not a colorimetric/CIE multi-primary solver. Dialing in a wide-gamut RGBY/RGBCV projector is a by-eye calibration, not an automatic computation. That's the crack the high-end rivals are praised for filling.

Why Lasergraph DSP earns the "best color" reputation inferred from specs

  • ~48-bit color (16-bit/channel, "281 trillion colors") — fades and low-level atmospheric beam dimming barely band.
  • Native 6-color driving real 577 yellow / 590 orange / 488 cyan on dedicated channels, not RGB approximation.
  • Per-laser bias / soft-blank offset + per-channel white balance — clean blacks, accurate white, correct thin-line color, per projector.
  • Real-time palette engine + up to 250 kHz color modulation. Priced ~$5k+, aimed at deep programmers.
Premise corrections. Several "everybody says" items are wrong: Phoenix / Showcontroller is now Pangolin-owned (a tool once cited as beating Pangolin). "MediaLas LSX" conflates two products — LSX is DrLava/Innolasers; MediaLas makes the RGB-only Mamba. There is no Coherent show software — "Lasershow Designer" (LD2000) is Pangolin's; Coherent makes hardware. The genuine multi-primary rival is Lasergraph DSP.

Comparative "better than Pangolin" claims are largely Photonlexicon forum opinion — those threads block scraping, so no quantitative head-to-head was verifiable.

05

Color science for aerial beams

This is the part that actually informs "the best possible beam projector." Beams are not graphics: you optimize per-wavelength luminous punch, saturation, atmospheric scattering, and beam divergence — none of which the flat CIE chromaticity triangle captures.

Why add a 4th color to a beam rig

  • Raw lumens — a dedicated yellow (~577–593 nm) sits near the eye's sensitivity peak, adding brightness without "spending" red + green to synthesize it.
  • Neutral true-white — three narrow spikes hit white on paper but read tinted in air; a 4th wavelength fills the spectral gap and makes warm/cool white tunable.
  • Clean secondaries — mixed yellow skews orange/dirty and mixed cyan is weak; dedicated lines fix that.
  • Gamut is the weakest motive for beams — a beam is one saturated color, and area ≠ brightness.

Scattering — the nuance most people get half-right established physics

Rayleigh scattering ∝ 1/λ⁴ (blue scatters far more) governs fine haze / clean air — so a blue beam does look disproportionately bright from the side there. But dense theatrical fog is the Mie regime (droplets ≈ or > wavelength), which is roughly wavelength-independent — that's why fog looks white-grey. So "blue beams look brightest because they scatter most" is true in light haze and overstated in fog; in a typical hazy venue, side-view brightness tracks eye sensitivity, and green wins.

Neutral white is hard for a reason you already modeled

Because green is ~3× more eye-efficient, a neutral white needs more red and blue optical power than green — the tool's "limiting line" logic. And critically, the colors' beam diameter and divergence must match, or the white only overlaps perfectly at one distance and fringes / tints elsewhere. That is exactly the tool's beam-corrected mode and halo warning — the literature arrives at the same feature independently.

Helmholtz–Kohlrausch. Saturated blue and red beams appear brighter than an equal-luminance white — strongest in the dark, where beam shows live. Another punch the flat gamut plot hides.
06

What it means for d65_mixer

The show packages do runtime color rendering on a projector you already own. This tool answers the upstream question none of them do: which lasers, at what wavelengths and power, to buy for a neutral white beam and a good palette. The research points a beam-oriented version toward:

  • Weight white/brightness by luminous efficiency — surface lumens / relative punch per line (the tool already carries the ȳ curve). This is what gamut hides.
  • Keep the beam-geometry / spot-overlap model — the research calls this the underrated real-world factor, and it's already built.
  • Model the real 6-channel target (R,G,B + DeepBlue/Yellow/Cyan — the FB4/DSP reality) so a design maps cleanly onto hardware.
  • Optionally flag Helmholtz–Kohlrausch punch for saturated beams.
